# Publishing Gemba Filesend to Google Play

This is the Android side of the app. It's a **Trusted Web Activity
(TWA)** that wraps the existing web app at
`https://gemba-filesend.vercel.app` — same encryption, same uploads,
same downloads, same UI. Code changes flow to the app automatically the
moment a new deploy reaches production.

## Why TWA and not native Kotlin?

The whole upload flow lives in the browser already: AES-128-GCM via
Web Crypto, presigned-URL uploads to Vercel Blob, two-step download
with client-side decrypt. Re-implementing all of that in Kotlin would
double the maintenance surface for an app that has to behave exactly
the same as the web. A TWA gives us a real Android app that runs
full-screen, ships through Play, and passes Play review **as long as
the web app meets PWA install criteria** — which the work in this PR
does.

## Repository layout

```
android/
  twa-manifest.json       # Bubblewrap source of truth
  README.md               # local build instructions
public/
  manifest.webmanifest    # PWA manifest the TWA reads from
  icon-*.png              # 192/512, plus maskable variants
  apple-touch-icon.png    # iOS home-screen
  play-store-icon.png     # 512x512 for the Play listing
  sw.js                   # service worker (installability + shell cache)
  .well-known/
    assetlinks.json       # Digital Asset Links — verifies the TWA owns the domain
scripts/
  generate-icons.mjs      # regenerates the rasters from the SVG sources
```

## Prerequisites

- Google Play Console developer account ($25 one-time).
- A computer with Node 18+, Bubblewrap (`npm i -g @bubblewrap/cli`),
  and Java 17 (Bubblewrap installs the JDK on first run).
- Push access to the `gemba-filesend.vercel.app` deployment (you need
  to update `assetlinks.json` between builds).

## Step 1 — Confirm the web app is install-ready

In Chrome on Android, open `https://gemba-filesend.vercel.app` and use
Lighthouse → **Installable**. You should see:

- ✓ Web app manifest meets installability requirements
- ✓ Service worker is registered

Both are handled by this PR: `public/manifest.webmanifest`,
`public/sw.js`, and the registration script in `src/app/layout.tsx`.

## Step 2 — Build the AAB

```bash
cd android
bubblewrap init --manifest=https://gemba-filesend.vercel.app/manifest.webmanifest
# The init flow asks for the package name — accept `mba.ge.filesend`.
# Bubblewrap then generates a Gradle project and prompts for keystore.

bubblewrap build
```

Output:

```
app-release-bundle.aab    ← upload this to Play Console
app-release-signed.apk    ← optional, for adb install testing
```

The keystore (`android.keystore`) is created on first build and is
already in `.gitignore`. **Back it up immediately** — losing it means
losing the ability to update this app on Play. Use a password manager
for the key passwords.

## Step 3 — Wire up Digital Asset Links

The TWA only runs full-screen if Chrome can verify, via Digital Asset
Links, that the Android app and the web host belong to the same owner.

1. Get the SHA-256 fingerprint of the upload key:
   ```bash
   keytool -list -v -keystore android/android.keystore -alias android
   # Copy the line that starts with "SHA-256:" — keep the colons.
   ```
2. Open `public/.well-known/assetlinks.json` and paste the fingerprint
   into the `sha256_cert_fingerprints` array.
3. Deploy. Verify the file is reachable at
   `https://gemba-filesend.vercel.app/.well-known/assetlinks.json` with
   `Content-Type: application/json`.
4. Test via Google's verifier:
   `https://digitalassetlinks.googleapis.com/v1/statements:list?source.web.site=https://gemba-filesend.vercel.app&relation=delegate_permission/common.handle_all_urls`

## Step 4 — Create the Play Console app

In Play Console → **Create app**:

- App name: **Gemba Filesend**
- Default language: English (US)
- App or game: App
- Free or paid: Free
- Declarations: tick the developer programme & US export laws boxes
  (no other declarations apply — no ads, no IAP, no kids).

## Step 5 — Fill the store listing

| Field | Value |
|---|---|
| Short description | Secure, end-to-end encrypted file sharing. Upload a file, share a link, the recipient downloads in-browser. |
| Full description | (see snippet below) |
| App icon | `public/play-store-icon.png` (512×512, regenerated by `scripts/generate-icons.mjs`) |
| Feature graphic | 1024×500 — design separately, brand-coloured |
| Phone screenshots | At least 2, max 8, 16:9 or 9:16. Pre-generated under `play-screenshots/` (1080×2400 portrait, 6 shots covering home + upload + download in light and dark themes). Regenerate with `node scripts/play-screenshots.mjs` against a running dev server. |
| Category | Tools (Productivity also acceptable) |
| Contact email | Use a real, monitored address. |

## Step 6 — Privacy policy

Play requires a hosted privacy policy URL. Add a `/privacy` page to the
web app (out of scope of this PR — quick follow-up) covering:

- What's collected: encrypted file blobs + minimal metadata (filename,
  size, optional salted password hash, counter, expiry).
- Where it's stored: Vercel Blob, EU/US, deleted automatically when
  expired/exhausted or after 24h via the cleanup cron.
- What's NOT collected: encryption keys (they live in the URL
  fragment and never reach the server).
- Contact email.

## Step 7 — Data safety declarations

In Play Console → **App content → Data safety**:

- Does your app collect or share any user data? **Yes** (file content +
  filename, while in transit and at rest).
- Data type: **Files and docs**.
- Collected: **Yes**.
- Shared with third parties: **No**.
- Encrypted in transit: **Yes** (HTTPS).
- User can request deletion: **Yes** — every share auto-deletes; the
  user can manually exhaust the counter or wait for expiry.

## Step 8 — Content rating questionnaire

Pick **IARC** → category **Utility/Productivity** → answer "No" to
every content question. The output will be PEGI 3 / Everyone.

## Step 9 — Target audience and content

- Age group: 13+
- Ads: No
- COPPA: not applicable

## Step 10 — Internal testing track first

Don't push to production immediately. In Play Console → **Testing →
Internal testing**:

1. Create a release with the AAB from `android/`.
2. Add your own email as a tester.
3. Wait for review (typically a few hours on internal track).
4. Install the test build on your phone via the opt-in URL Play gives
   you.
5. **Confirm there is no Chrome address bar across the top** of the
   running app. If there is, asset links failed — go back to step 3.
6. Run through upload (single + multi-file), download, password,
   exhausted link, expired link.

## Step 11 — Add the Play app-signing key to assetlinks

Once Play accepts the AAB, it generates its own *app signing key*
(distinct from the upload key you built with). The user-facing APK is
re-signed with that key, and Chrome verifies against **that** key's
fingerprint — not your upload key's.

1. Play Console → **Setup → App signing** → copy the SHA-256
   fingerprint of "App signing key certificate".
2. Add it as a second entry to `public/.well-known/assetlinks.json`:
   ```json
   "sha256_cert_fingerprints": [
     "AB:CD:...:upload-key",
     "EF:GH:...:play-app-signing-key"
   ]
   ```
3. Deploy.
4. Reinstall from the Play internal-test track and verify
   full-screen again.

## Step 12 — Promote to production

Closed testing → Open testing → Production. Plan for ~7-day review on
the first production submission.

## Updating the app later

For most changes (UI, behaviour, bug fixes) you don't need a new AAB —
just deploy the web app. The TWA pulls the latest version automatically.

You only need to rebuild and re-upload the AAB when:

- The package name, app name, theme color, icon, or shortcut list
  changes (anything baked into the AAB).
- You bump `appVersionCode` and `appVersionName` in
  `android/twa-manifest.json` and re-run `bubblewrap build`.
- Google bumps a Bubblewrap or Play target SDK requirement.

## Regenerating icons

Edit `public/icon.svg` / `public/icon-maskable.svg`, then:

```bash
node scripts/generate-icons.mjs
```

That writes all the PNGs the manifest, the TWA, and Play need.
